## 4. 변경 배경 및 결과

### 배경 (Why)

- 디자인 시스템(`design.md`)과 22 화면 와이어프레임(`wireframe.md`)이 완성된 상태 — 코드 구현 시작 시점.
- React 기반 클릭 가능 프로토타입을 22 화면 모두 demo 가능 수준까지 가져갈 계획 (`plan.md` + `sprints/`).
- S1은 4개 sprint 중 첫 번째로, 후속 sprint들이 의존하는 "뼈대"를 만드는 단계.
- 디자인 시스템 토큰을 코드로 옮기되 **단일 출처(single source)**를 유지해 후속 디자인 변경 비용을 최소화하는 것이 핵심 목표.

### 결과 (What)

- `npm run dev` → 모든 22 라우트 진입 가능 (placeholder 또는 실제 컴포넌트)
- `npm run build` → 17 lazy chunks + main 70KB gzip (코드 분할 D7 동작)
- `npm test` → vitest + RTL 환경 동작 (1 dummy 통과)
- `npm run typecheck` → 0 에러 (strict mode + path alias)
- `/dev/components` → 12 UI primitive 모든 variant 시각 검증 가능
- 4탭 BottomNav 네비게이션 + Header + TabLayout 동작
- design.md §8 토큰이 `src/tokens.ts`에서 단 한 곳만 정의되어 Tailwind와 코드가 동일 값 사용

### 추가 polish 사이클 (836a48a)

S1 첫 commit 후 직접 시각 검증한 결과 quality 부족 — 사용자 피드백("퀄리티가 낮은 거 같은데")으로 inline 디자인 리뷰 1라운드 진행:
- 캘린더 셀: 동그라미 → **사각형** (사용자 직접 요청, design.md §5.8 정합 개선)
- Color showcase: 작은 점 + 텍스트 겹침 → 2-col 카드 grid
- TodayCard / RecommendCard: 시각 위계 강화 (그린 액센트 / 노란 추천 배지)
- 입력·버튼·셀렉터 전반: 1.5px border + active:scale 피드백 + transition

## 5. 중요 변경 사항

### 아키텍처

- **단일 AppContext** (D3): provider 1개로 user / phq9 / preference / challenges / posts 모두 관리. zustand 도입은 prototype 규모엔 과함 — 후속 sprint에서 필요 시 교체.
- **Layout route 패턴**: 4탭(`/home`, `/diary`, `/cert`, `/my`)이 `<TabLayout>` parent route를 공유 — Header + BottomNav가 코드 중복 없이 4탭에만 노출.
- **Wizard 단일 라우트** (D2): `/check/quiz` (PHQ-9 + 선호조사 14 step), `/create/new` (5 step) — URL param 대신 컴포넌트 내부 step state 사용. back-button 안전, deep link 단순.

### 토큰 단일 출처 (D5)

- `src/tokens.ts`만 수정하면 Tailwind class와 컴포넌트 인라인 style이 동시에 갱신.
- `tailwind.config.ts`가 TypeScript 파일이라 import 깔끔. (S1 도중 typecheck 에러로 `tsconfig.node.json`에 tokens.ts include 추가, `as const` 일부 제거로 Tailwind 타입 호환.)

### 코드 분할 (D7)

- 17 화면 모두 `React.lazy()` 적용.
- `Loading` 컴포넌트만 eager import (Suspense fallback이 lazy면 chicken-and-egg flicker — S1-A2).
- 결과: initial JS bundle 70KB gzip / 화면별 lazy chunk 평균 0.3KB (placeholder 단계).

### Card 컴포넌트 4 분리 (D-S1.2)

- `<Card variant="...">` 단일 컴포넌트 대신 `TodayCard` / `FeedCard` / `RecommendCard` / `AddCard` 4개 named export.
- 각 variant의 데이터 shape이 달라 prop drilling이 어색해지는 것을 회피.
- 같은 `Card.tsx` 파일에 co-locate해 발견성은 유지.

### Premature abstraction 회피 (D-S1.1)

- S1 sprint plan에 `FormField` + `StepActions` 헬퍼 작성 task가 있었으나 **삭제**.
- 실제 소비처(Signup, Steps wizard)가 S2/S3에서 만들어지므로 사용 패턴이 확정되지 않은 상태에서 만들면 후속에서 props 깨짐.
- Rule of 3 적용해 S3 시점에 추출.

### Quality polish (836a48a 통합)

- Calendar 셀: `rounded-full` → `rounded-md` (사용자 직접 요청, 마커 → 캘린더 그리드)
- 모든 인터랙티브 요소에 `active:scale-[0.97]` press 피드백 + `transition-all duration-150`
- 모든 border 1px → 1.5px (시각 정의력 강화)
- 모든 input focus state border-primary
- TodayCard 좌측 그린 액센트 strip + Sparkles 아이콘 + pill-style 완수 배지
- RecommendCard 우상단 노란 "추천" 배지 (design.md §5.3 spec 준수)
- AddCard 점선 border + 원형 + 아이콘 (interactive 신호)
- Loading: animate-pulse → animate-bounce stagger (작동 중 느낌)

## 7. 요약

ShallWe React 프로토타입의 첫 코드 단계입니다. 디자이너가 정의한 색상·폰트·로고가 화면에 실제로 보이게 만드는 "기초 공사" 작업으로, 사용자가 보는 결과물은 아직 placeholder가 대부분이지만 다음 4가지가 동작합니다:

1. **로고 + 하단 네비게이션 바** — 다이어리 / 홈 / 인증 / 마이 4개 탭 클릭 시 화면 전환, 활성 탭이 그린으로 강조됨
2. **디자인 시스템 12 컴포넌트 미리보기** — `/dev/components` 페이지에서 버튼·입력창·카드·캘린더 등이 정의된 색·폰트·여백 그대로 보이는지 한 화면에 모아 검증 가능
3. **앱 컨테이너** — 모든 화면이 모바일(440pt) 폭 안에 정렬되고, 데스크탑 브라우저에서도 모바일 비율 그대로 가운데 정렬
4. **개발 환경** — 코드 변경 즉시 화면 반영(HMR), 자동 테스트 실행, 빌드 산출물 분석 모두 동작

Sprint S1의 "끝까지 가면 보이는 모습"을 그대로 달성했고, 다음 Sprint S2(회원가입 → 무기력 검사 → 결과)에서 이 기초 위에 실제 사용자 플로우를 얹습니다.

본 commit log는 사용자 피드백을 받아 디자인 quality 1라운드 보완(캘린더 셀 사각형 + 카드 시각 위계 강화 + 입력 요소 정밀도)까지 포함합니다.
